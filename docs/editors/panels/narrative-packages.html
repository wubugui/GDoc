章节导演

章节导演面板在主编辑器的 叙事编排 分类下。你用它管理「叙事包」：哪一包剧情应该挂在哪个场景里，什么时候活起来，什么时候算做完。它不负责演出本身，不写对白，不决定角色怎么走位，也不替你编一段过场；它只管一件很关键的事：在某个场景里，哪些叙事包处于 live，哪些叙事包处于 dormant。

如果你把雾津的一段剧情想成戏班子上台，叙事状态机像总场记，知道现在演到第几折；图对话像台词本，负责每个人说什么；场景进入时的动作像开场锣，玩家一进门就敲一下；章节导演则像候场牌，告诉系统「茶馆开场包现在能上」「背尸线包先候着」「阎王岭包等进度够了再亮」。读完这页，你会知道怎么把这些包排进场景里，怎么检查激活与完成条件，以及什么时候该交给别的面板处理。

这是什么​

章节导演管的是「叙事包清单」。每一条叙事包记录都围绕五件事：

  • 标识 id：给作者和系统认这一条用。它要稳定、好认，别今天叫茶馆开场，明天又改成说书起头。
  • 绑定场景：这一包在哪个场景里生效，例如茶馆、雾津街头、义庄门口。
  • 绑定叙事包或图：这一包真正接到哪段叙事内容，可能是一组已经整理好的叙事包，也可能是某张图对话。
  • 激活条件 when：什么时候从 dormant 变成 live。比如玩家已经听过茶馆评书，或某条主线推进到码头以后。
  • 完成条件 done：什么时候这包不再需要待命。比如开场对话播过、背尸交付完成、阎王岭入口已经揭开。

它的边界很清楚：章节导演只判断「哪一包活」。包里具体有多少节点、对白怎么分支、状态怎么流转，是叙事状态机和图对话的事。你在章节导演里不应该试图复刻一整段演出逻辑；那会让包外包内各管一半，后面很难查错。

雾津的典型用法是按进度换包：开局茶馆里先 live 的是茶馆开场包，玩家听书、拿到第一层线索后，背尸线包开始在义庄和街头待命；等码头洋人线把危险感推高，阎王岭包才在山口附近活起来。这样同一个场景可以随着故事阶段换出不同的一组对话、热区和提示，而不必把所有判断塞进一个巨大的场景入口。

入门手把手​

第一步：打开面板​

启动主编辑器后，沿左侧导航进入 叙事编排 → 章节导演。如果你是第一次来，先不要急着新建很多条；选一个场景，确认现有列表里有没有已经挂着的叙事包，理解它们各自是为了哪段剧情服务。

第二步：新建一条叙事包记录​

点新建后，先填一个稳定的标识。建议你用剧情含义来命名，而不是用「测试」「临时」「新包」这类会过期的叫法。然后选择绑定场景。绑定场景不是「玩家最终会去哪里」的概念，而是「这一包需要在哪个场景里被章节导演判断」。茶馆开场包就绑茶馆，背尸线包可能绑雾津街头、义庄，阎王岭包则绑山口或相关入口场景。

第三步：绑定叙事包或图​

接着把这条记录接到你已经准备好的叙事包或图对话。这里不是写内容的地方，只是建立关系。若下拉或选择器里找不到目标，先回到对应的叙事内容面板，把包或图整理好再回来接。

第四步：填写激活条件​

激活条件决定这包什么时候 live。新手最稳的写法是用清晰的剧情进度做门槛：完成茶馆听书后，茶馆后续包才激活；接下背尸活计后，背尸线包才激活；码头洋人线出现关键进展后，阎王岭包才激活。

条件不要写得太隐蔽。你半年后回来，应当一眼能看出这包为什么会活。

第五步：填写完成条件​

完成条件决定这包什么时候 dormant。茶馆开场包可以在开场对白结束后完成；背尸线包可以在尸体送到义庄、交差完成后归档；阎王岭包可以在玩家真正进入山口后的第一段引导完成时休眠。

完成条件不是奖励，也不是下一段剧情。它只是告诉章节导演「这一包的候场使命结束了」。奖励、开地图点、给任务、发信号这些事，仍然放在任务、场景进入时动作、图对话或叙事状态机里。

第六步：Apply 和保存​

填完后先点 Apply。Apply 会把当前表单写回叙事包列表，也就是 packages 列表。随后再执行保存，让变更真正落到项目数据里。只改了表单但没有 Apply，或 Apply 后忘记保存，预览里都可能看不到你刚才的安排。

最后进入预览，从对应进度走一次：茶馆开场包是否只在开局活跃，背尸线包是否在接活后才出现，阎王岭包是否没有提前露头。确认这些，再继续扩更多包。

进阶可填项与配合​

标识与命名​

标识要服务于排查。你看到它时，应当能马上知道这是哪一段、在哪个场景、为什么存在。避免用角色名单独命名，因为同一个角色可能横跨多条剧情线；也避免用纯序号，因为序号在重排章节后会失去意义。

绑定场景​

绑定场景是章节导演的工作范围。它不会替你把玩家传送到场景，也不会自动生成场景里的热区。你需要在场景面板里准备好进入路径、交互点、进入时动作。

如果同一段剧情跨多个场景，你可以拆成几条叙事包记录，各自绑定对应场景。比如背尸线在雾津街头负责接活，在义庄负责交付，在城隍庙附近负责听到旁人传闻。这样每个场景只背自己该背的包，排查时更清楚。

绑定叙事包或图​

叙事包适合装一组相关内容，图对话适合直接接一张对话流程。你要根据剧情规模选择：一段短的茶馆问答可以直接绑图；一条会跨场景、跨条件反复出现的背尸线，更适合整理成包再绑。

激活条件 when​

激活条件常和叙事状态机、任务状态、旗标、场景进入时动作配合。比如叙事状态机推进到「码头疑云」后，码头洋人线相关包才 live；玩家接下背尸活计后，义庄的交付包才 live；茶馆听书完成后，街头路人的传闻包才 live。

写激活条件时，你要问自己三个问题：玩家是否能自然满足这个条件；如果条件不满足，场景里是否还有合理内容可玩；如果条件提前满足，会不会泄露后面的剧情。只要其中一个答案不稳，就先回到状态机或任务面板把进度门槛整理清楚。

完成条件 done​

完成条件的重点是「退场」。如果一包没有完成条件，它可能会一直 live，导致玩家后期回茶馆还反复撞到开场内容。相反，如果完成条件太早，玩家可能还没看完关键线索，这包就休眠了。

你可以把完成条件设在玩家完成一次关键交互之后，也可以设在某个叙事状态进入之后。比如阎王岭山口包在玩家第一次听到山风里的提示后仍然保留，直到真正进入岭口引导完成再休眠，这样比「刚解锁就完成」更稳。

和叙事状态机配合​

章节导演决定哪一包活，叙事状态机管包内外的状态流转。最常见的配合是：状态机进入某阶段，激活条件满足；玩家在包内完成对话或交互后，包内动作发信号；状态机再根据这个信号推进到下一阶段。

你不要让章节导演替状态机决定故事已经走到哪一幕。章节导演可以读进度，但不应该成为进度本身。否则同一条主线既被状态机管，又被包的激活条件暗中管，后面会出现「包活着，但故事阶段没到」或「故事到了，包却没活」的错位。

和场景 onEnter 配合​

场景进入时可以作为接线点。玩家进入茶馆时，场景先执行进入时动作，再由章节导演判断当前 live 的叙事包，图对话或热区再根据 live 包展示对应内容。你可以把场景 onEnter 当成「敲门」，章节导演当成「开哪间屋」，图对话当成「屋里说什么」。

和图对话配合​

图对话适合处理包内细节：分支对白、玩家选择、线索记录、是否发信号。章节导演只需要知道这张图是否应该在当前场景里候场。茶馆开场包可以接一张说书图，背尸线包可以接一组街头与义庄对话，阎王岭包可以接山口的引导图。

危险区与边界​

  • 不要把演出逻辑塞进章节导演。演出属于图对话、过场、动作和场景。
  • 不要让多个包在同一条件下同时 live，除非你明确想让它们并存。并存会增加玩家看到重复入口的概率。
  • 不要忘记完成条件。没有退场的包会像旧戏牌一样一直挂在门口。
  • 不要用临时标识。标识一旦被条件或其他内容引用，随意改名会让引用断开。
  • 不要把调试用的包留在正式进度里。玩家不该看到作者用来试流程的候场内容。
  • 不要指望 Apply 等于完整保存。Apply 写回列表，保存才让项目状态稳定下来。

章节导演也不负责地图解锁、任务奖励、物品发放、角色站位、滤镜切换。它可以和这些系统配合，但不替它们做决定。边界越清楚，后期越容易维护。

常见问题​

为什么我进茶馆没有看到新包？​

先检查三件事：这条记录是否绑在茶馆，激活条件是否已经满足，Apply 和保存是否都做了。若三者都对，再去图对话或场景热区看入口是否真的接到了这包。

为什么玩家后期还会看到开场内容？​

多半是完成条件没写，或完成条件永远不会满足。给茶馆开场包加上明确的退场条件，例如开场说书结束或第一次关键选择完成。

一个场景能不能有多个 live 包？​

可以，但要谨慎。茶馆里同时 live 茶馆后续包和码头传闻包是合理的，因为它们服务不同信息；如果两个包都抢同一个入口，玩家就会觉得内容重复或顺序混乱。

背尸线跨多个场景，该做一个大包还是拆开？​

如果每个场景的内容差异很大，建议拆开。雾津街头负责接活，义庄负责交付，城隍庙附近负责旁人传闻。这样条件更清楚，退场也更干净。

状态机和章节导演谁说了算？​

状态机说「故事走到哪」，章节导演说「这个场景里哪包活」。你可以让章节导演读取状态机进度，但不要反过来让包清单承担主线推进。
