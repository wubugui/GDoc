旗标面板

任务完成了没有、玩家有没有见过湿鞋、庙门开没开——这些进度事实通常记在旗标（flag）里。旗标面板（旗标注册表）不是改某个存档里某个旗标当前是真是假，而是声明这个工程里允许存在哪些旗标键、值是什么类型、动态 id 按什么规则拼。真正改玩家某次游玩里旗标的值，要去动作里用「设旗标」，或者由剧本在阶段推进时写入。读完这页你能建好一批旗标键，并搞清楚这个面板里有一整块「你在界面上看不到、改不到」的盲区,别在协作里假装自己能维护它。

这是什么（30 秒看懂）​

把旗标想成雾津剧情组的一份登记名册：名册上只列「有哪些进度事实可以被记录、每一项是什么类型（真假、数字……）」,不记录「某个具体玩家现在这项是真是假」。就像户籍簿只登记「本县允许登记的字段有姓名、年龄」，不代表某个人此刻多大——那是每份具体档案（也就是玩家存档）自己的事。旗标面板管的是名册本身；名册之外，还藏着一块名册管理员自己维护、你翻不到的抽屉（迁移规则、运行时注册），那部分不归编辑器管，需要找程序处理。

入门：手把手做第一次​

以「渡口解锁」和一批章节标记为例，从零做一遍。

  1. 打开 ./dev.sh editor → 注册 → 旗标。
  2. 切到静态旗标表，点添加，key 填 dock_unlocked，值类型选「布尔」。
  3. 再添加几条常用的：clue_shoe（是否见过湿鞋线索）、temple_entered（是否进过庙）——命名尽量全项目统一风格（比如都用小写加下划线），别随手写 flag1 这种没有意义的名字，往后排查会很痛苦。
  4. 点 Apply 保存。
  5. 去剧本面板，在「渡口」这个阶段的 exposes 里，把 dock_unlocked 写成真值——这一步才是真正让某次游玩「触发」这个旗标的地方。
  6. 去地图面板，把渡口这个地点的解锁条件设成读取 dock_unlocked 这个旗标。
  7. 进入运行预览：在剧本阶段完成之前打开地图，确认渡口是锁着的；推进剧本让这个阶段完成之后再打开地图，确认渡口解锁了。

进阶：每一项都讲透​

静态旗标（static）​

  • key：旗标的名字，也就是别处引用它的唯一凭据。
  • 值类型：这个键存的是什么类型的值——常见是布尔（真/假），也可能是数字等其他类型，取决于你项目支持的类型选项。

静态旗标就是「一个键对应一件具体的进度事实」，适合数量不太多、语义各自独立的场景，比如「章节完成」「见过某条线索」「某扇门开没开」。

模式规则（pattern）​

当你需要「每个 NPC 一条『已对话过』旗标」「每个见闻条目一条『首次阅读过』旗标」这种批量重复的键时，一条条手写几十上百条 static 既繁琐又容易漏。模式规则就是为此设计的：定义一套前缀/后缀 + id 来源的拼接规则，比如「前缀 npc_talked_ + 来自 NPC 自身 id」，就能自动覆盖「和关二狗说过话」拼成 npc_talked_guan_ergou、「和李天狗说过话」拼成 npc_talked_li_tiangou，不需要为每个 NPC 单独登记一条 static。

模式规则特别适合：

  • 每个 NPC 一条「已对话」标记。
  • 每条见闻录/档案条目一条「首次阅读」标记。
  • 任何「同一类实体、同一种进度事实、只是具体对象不同」的场景。

条件/动作里怎么用旗标​

  • 在条件编辑器里选 flag 这一类叶子时，需要三样东西：key（从已注册的静态旗标或匹配某个模式规则的动态键里选）、比较符（等于、不等于、大于、小于等）、值（可以插入富文本标签）。
  • 在动作编辑器里，「设旗标」这个动作类型负责真正改变某次游玩里这个键的值，写入的 key 应该先在旗标面板注册过，不然协作流程/校验环节可能会报警。

改与删——语义变更的风险​

  • 改值类型非常危险：已经跑在存档里的旗标值，语义是跟着当初的类型定的，中途改类型可能让老存档里的旧值解读错乱。稳妥的做法通常是只加新键，不改旧键的类型，需要迁移语义就另起一个新 key。
  • 删除静态键：删之前要搜一遍所有引用它的条件、剧本的 exposes、任务相关判断、地图的解锁条件——删掉之后这些引用都会失效。
  • 旗标和任务自身的完成状态经常有重复语义的风险——比如「任务已完成」这件事，有的地方会额外用一个旗标去记录、有的地方直接判断任务状态。两者若同时存在，记得对好表，避免同一件事被两套机制分别记录、彼此不同步。

盲区——迁移块与运行时块（重要）​

旗标面板背后的注册数据里，还包含迁移块和运行时块这类结构：

  • 这些结构在编辑器界面上完全没有入口——你打开旗标面板，翻遍每个 Tab，都不会看到它们。
  • 它们会在数据里原样保留（不会因为你在面板里编辑保存别的部分而被抹掉），但人类没办法通过 GUI 去看它、改它。
  • 如果你的工作流需要新增或调整迁移规则、运行时注册这类东西，这已经超出了编辑器能协作到的范围——正确做法是找程序升级编辑器支持，或者走专门的处理流程，**而不是假装「在旗标面板里已经维护好了」**去策划会上汇报，也不要自己手写这块数据指望它被正确识别。

一句话记住：旗标面板能让你放心管的，是静态旗标表和模式规则；迁移与运行时这两块，看不见就是看不见，别硬来。

效率与配合技巧​

  • 全项目统一命名风格（比如都用 snake_case 或统一的点分层级），下拉搜索和排查问题都会顺手很多。
  • 开局就需要用到的旗标（游戏刚启动就该有默认值的那种），记得同步去全局配置里对应的开局旗标设置里也登记一下，两处要保持一致。
  • 涉及大量重复实体（NPC、档案条目）的进度标记，优先考虑模式规则，而不是手写一大堆看起来雷同的 static。

危险区与边界​

  • 迁移块、运行时块是彻底的盲区：编辑器完全不暴露，原样保留但人类改不到，需要这类变更请找程序处理，不要在协作中假装编辑器已经覆盖了这部分。
  • 改值类型极其危险：会改变已有存档里旗标值的语义，通常应该只加新键、不改旧键类型。
  • 删除静态键前必须全局排查引用：条件、剧本 exposes、任务判断、地图解锁条件都可能还在用这个 key。
  • 未注册就直接设旗标：调试时可能表面看起来有效，但正式协作/校验流程可能会报警，正规做法是先在旗标面板注册,再在动作里使用。
  • 更系统的「这个面板哪里改了会丢、哪里编辑器根本够不到」，参见危险区和可编辑面·危险区参考。

常见问题​

现象原因怎么办
条件编辑器的 flag 下拉里找不到这个键这个键还没在静态旗标表里注册去旗标面板添加对应 static 条目
存档里旗标语义莫名变了值类型被改动过尽量避免改类型，需要变化就另开新键并做数据迁移
觉得该有的迁移没生效迁移块是编辑器盲区，界面改不到找程序处理，不要自己在界面上找这个入口
设旗标动作报警/被拦这个键还没有在旗标面板注册就直接用了先注册 static 或确认匹配某条模式规则，再回动作里使用
地图/条件该解锁却没解锁旗标名字和实际写入的键不一致（拼写、大小写等）统一命名并核对剧本 exposes 与条件里用的 key 是否完全一致

预览验证​

  1. 注册好静态旗标或模式规则，Apply 保存。
  2. 在剧本的 exposes 或某个动作里写入这个键的值。
  3. 运行预览，触发这次写入。
  4. 用条件编辑器选同一个 key，分别测试真/假两种情况下判断是否符合预期。
  5. 读一次存档，确认这个值持久化了下来。
  6. 如果用的是模式规则，挑一个具体实例（比如某个 NPC）测试拼出来的动态键确实按预期生成。
