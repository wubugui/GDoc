# 文本库面板

界面上的「确认」「背包已满」、系统菜单文字、各面板反复要用的短句——都归**文本库**（strings）管。读完这页你能：新建一个键并让 UI 显示它、看懂"键改了值会全站联动"是怎么回事、也知道键和分类该怎么删。

---

## 这是什么（30 秒看懂）

如果把游戏里所有"要显示给玩家看的字"分个类，档案面板管的是长篇故事，图对话管的是当场说的台词，而**文本库管的是那些散落在各处、反复复用的短句**——按钮上的字、提示语、系统菜单。

打个雾津的比方：文本库像衙门里挂在各处的**匾额和告示牌**——"内有恶犬""此路不通"这种简短、到处都要用、还得保持一致的字，不属于哪一段具体的故事，但游戏运转起来处处都在读它们。

结构很简单：一棵**分类树**，树下挂**键值对**。键（key）是给程序认的名字，值（value）是玩家真正看到的文字，值可以是纯文本（还能走富文本）、数字、或者true/false这种开关量。

## 入门：手把手做第一次

最常做的事：新建一个键，填上文案，让某处 UI 显示出来。

```mermaid
flowchart LR
  A[打开主编辑器] --> B[资源 -> 文本库]
  B --> C[左侧树选/建分类]
  C --> D[添加键]
  D --> E[填值]
  E --> F[Apply 保存]
  F --> G[预览里打开对应 UI 查看]
```

1. 打开主编辑器：`资源 → 文本库`。
2. 左侧是分类树，选一个合适的分类（比如 UI 通用类），或者先想清楚这个键该归到哪一类。
3. 点**添加键**，键名建议用点分层级的写法，比如「ui.quest.tracked」，一看就懂它是 UI 里跟踪任务的文案。
4. 在值那一栏填「当前任务」这样的文案；如果这句话里要引用别的东西（比如玩家名字），可以用富文本标签。
5. 点 **Apply** 保存。
6. **验证**：运行预览，打开真正会读这个键的界面（比如地图标题、某个提示条），确认文字显示正确，没有出现空白或英文兜底文案。

雾津小例子：地图标题键「雾津」建议就填两个字、不加标点，和主菜单其它标题风格保持一致；改完直接在预览里打开地图看一眼，比光看编辑器里的值更保险。

## 进阶：每一项都讲透

### 分类树

分类树只是用来**组织键名**、方便你自己和团队找东西，不影响游戏逻辑本身。键该放哪个分类，取决于它属于哪块系统（比如"UI 通用""任务提示""压力/险境提示"）。分类越细，团队协作时越不容易互相踩键、也越方便本地化团队按批次导出翻译。

### 键值对

- **键（key）**：给程序/其它面板认的名字，本身不会显示给玩家看。命名建议保持稳定——一旦定下就尽量别改，因为改键名在这个面板里等价于"新建一个新键 + 自己去把所有引用它的地方手动改过来"，**面板不会帮你自动搬家**，旧键会孤零零地留在树里没人用。
- **值（value）**：玩家实际看到的内容，随类型不同走不同的编辑方式：
  - **字符串（str）**：走富文本，可以插名字、物品、旗标这些标签，也可以引用文本库里的别的键（形成"文案套文案"的复用）。
  - **数字（number）**：走数值输入。
  - **布尔（bool）**：走开关输入，一般用来控制某个系统提示的显示与否。

### 谁在用文本库

游戏里的 UI 元素、系统菜单会直接读这里的键（这一步是工程接线好的，策划这边只管把值填对）；此外，图对话、档案这些富文本框也可以用字符串引用标签去引用文本库里的一个键，把一句反复出现的话集中改一处、处处生效。

### 改值的影响范围

**改一个键的值会立刻影响所有引用它的地方**——这既是它的优点（改一处全站同步），也是它的风险（改错一处全站遭殃）。改值之前，养成先想一下"这个键还有谁在用"的习惯，避免动了一个通用键、结果好几处 UI 一起变了样。

### 删除（能删）

| 操作 | 能不能做 |
|---|---|
| 改值 | 可以，随时改 |
| 新建键 | 可以 |
| **删键** | **可以** —— 选中键之后，删除按钮、右键菜单、Delete 键都能删 |
| **删分类** | **可以** —— 删掉一个分类，这个分类底下挂着的键会跟着一起被删掉，动手前想清楚这个分类下有没有别处还在用的键 |

删之前记得先想一下这个键有没有别处在引用（参见上面「改值的影响范围」），免得删完某处 UI 突然找不到文案。

如果某个键的值是数组结构，保存后会被压成一整段字符串。需要存数组结构的数据，换别的地方维护那条数据，别用这个面板编辑它。

### 和别的面板怎么配合

- 和[富文本](../concepts/rich-text)配合：字符串引用标签就是从这里引一个键出来，用在对话、档案的正文里。
- 和[全局配置](./config)配合：不少系统层面的文案本身就是走文本库的键，改系统提示先来这里找。
- 和[档案](./archive)分工：文本库放**短句**，长篇剧情文字应该放档案或图对话，别把大段说明文字硬塞进一个键里维护。

### 效率窍门

- 键名用点分层级（比如「ui.地区.用途」）比拍脑袋起名更好维护，团队一眼能看出这个键归哪块系统管。
- 批量新增的时候，建议先在文档里列好这一批键名和对应文案，再逐条录入，比边想边填更不容易漏。
- 长期不用的键，与其放着不管，不如定期跟程序对一轮，清理掉真正废弃的部分，避免分类树越滚越大。

## 危险区与边界

| 边界 | 说明 |
|---|---|
| **键、分类都能删** | 删除按钮、右键菜单、Delete 键都能删；删分类会连带删掉底下挂着的键，动手前想清楚 |
| **改键名不会自动搬家** | 等于新建键+自己手动改所有引用处，旧键留在树里，要清理得自己手动删 |
| **数组值会被压成字符串** | 需要存数组结构的数据，换别处维护，不要用这个面板编辑 |
| **改值全站联动** | 改之前先搜一下这个键还有谁在引用 |
| **长文别塞进 strings** | 大段剧情文字应该放[档案](./archive)或[图对话](./dialogue-graph) |

更完整的编辑器风险说明，见[危险区](../concepts/danger-zone)。

## 常见问题

| 现象 | 原因 | 怎么办 |
|---|---|---|
| UI 显示一片空白 | 键名打错，或忘了 Apply | 核对键名拼写，确认已保存 |
| 改了一个键，好几处 UI 都变了样 | 多处 UI 共用同一个键 | 要么接受联动，要么拆成各自独立的键 |
| 过时的键越堆越多 | 没人去删而已，界面本身有删除入口 | 选中直接删，删之前先确认没别处还在引用这个键 |
| 富文本标签写得很复杂但显示乱 | 短句场景不适合太长的 markup | 把这段内容迁到档案或对话里 |
| 切换分类后发现刚编辑的值没了 | 忘了先 Apply 就切换了分类 | 养成先保存再切换的习惯 |
