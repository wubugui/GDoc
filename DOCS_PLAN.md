# GameDraft 文档撰写计划

> 这是**派活清单**：把整套用户文档拆成一页一任务，交给写手（agent）逐页写。
> 每页给出：路径、标题、**读者读完能做到什么**、**写手怎么学会它**、要点。
> 先读第 0 节「写作铁律」，违规的一律打回。

---

## 0. 写作铁律（每个写手必须遵守，违反即作废重写）

1. **只写"怎么用"，不写"怎么实现"。** 这是给**使用者**（策划/美术/玩家）看的，不是给程序员看的。
2. **禁止出现任何代码相关内容**：
   - ❌ 源文件名（如 `xxx_editor.py`）、模块名（如 `tools.xxx`）、类名、函数名、字段的内部结构
   - ❌ 代码块、伪代码、JSON 结构示例（除非那是**用户在界面里真的要填的内容**）
   - ❌ "权威源是代码""见代码""实现在……"这类话
   - ❌ 任何指向代码仓库/源文件的引用或链接
   - ✅ **唯一允许的"命令"**：用户为了打开/运行工具要亲手敲的操作命令（如 `打开主编辑器` 这一步）。能用"在控制台点某按钮"代替就优先用界面说法。
3. **第二人称、任务导向、大白话。** "你想让 NPC 说话，就……"，不是"NPC 对话由对话系统驱动"。
4. **每页必须配图**：
   - **实操截图**：打开对应工具，做一遍，截关键步骤（框出要点哪里）。
   - **示意图**：流程/概念用简单示意图讲清（可用 Mermaid 或内嵌 SVG）。
   - 没有截图的页面视为未完成。
5. **用游戏里的真实例子举例**：雾津的场景、关二狗的对白、城隍庙的规矩……比抽象例子亲切，也顺带展示游戏气质。
6. **术语第一次出现给一句大白话解释**，然后链到术语表。
7. **宁细勿漏**：把一件事讲透。这是个大项目，文档要配得上——一页只讲一件事，但那件事讲全。
8. **语气入戏**：延续"雾津折子"的调性（油灯、书案、编纂），别写成冷冰冰的规格书。

以下页面**目前塞满了源文件名/模块名/实现说明**，必须按铁律重写成纯用法（保留有用的用户信息，删掉一切代码引用）：

| 路径 | 现状问题 | 重写成 |
|---|---|---|
| `editors/main-editor/overview` | 列了每个面板的源文件名、类名 | 只讲"主编辑器能做什么、界面怎么逛、30 块面板各管什么活"（用面板的中文名） |
| `editors/main-editor/shared-*` | 讲了控件实现 | 改成"怎么编排一个动作/设一个条件/写带引用的文本" |
| `editors/*/[每个工具页]` | 全部带"模块 `tools.xxx`"速览表 | 去掉模块行，改"这工具帮你做什么 + 怎么开 + 一步步用" |
| `reference/authoring-surface` | 提了一堆内部字段/结构 | 保留"哪里安全/哪里会丢/哪里够不着"的**用户后果**，删掉字段内部结构与文件名 |
| `editors/launch-architecture` | 讲了三层启动链、映射表（纯实现） | **整页删除或改为**"每个工具用什么命令/按钮打开"的纯操作速查 |
| `dev/architecture` | 双壳、命令清单 | 移出用户文档；开发向内容单独放，不混进使用手册 |

> 原则：**使用手册区（教程/编辑器手册/参考/玩家手册）里不允许有任何实现内容。** 开发向文档（架构、移植）单独成区，且也尽量讲"怎么用工作流"而非"代码长啥样"。
